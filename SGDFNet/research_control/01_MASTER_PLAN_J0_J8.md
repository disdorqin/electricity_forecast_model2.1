# SGDFNet Master Plan J0-J8

This roadmap is the decision-complete execution contract for the next SGDFNet research cycle. Each stage uses the same schema and must be completed in order.

## J0: Protocol and Ledger Lock

- Purpose:
  - lock Protocol B as the primary 2025 rolling monthly evaluator
  - register historical A0, A1, A3, and A3R evidence
  - establish baseline definitions, ledger discipline, and best-model registry
- Allowed changes:
  - evaluator design documents
  - baseline registration scripts or records inside SGDFNet-owned paths
  - artifact schema and control metadata
- Forbidden changes:
  - new model modules
  - result-driven redefinition of metrics
  - any protected-directory edits
- Experiment variants:
  - protocol lock audit
  - historical baseline registration
  - metric consistency audit
- Per-module tuning loop:
  - none; this stage is documentation and registration only
- Pass criteria:
  - Protocol B is fixed
  - baseline definitions are explicit
  - ledger and registry are initialized
- Fail criteria:
  - protocol ambiguity remains
  - historical baselines are not traceable
- Switch criteria:
  - not applicable; complete J0 before moving on
- Output artifacts:
  - control docs
  - initialized ledger
  - initialized best-model registry
  - state file
- Advance condition:
  - J0 pass criteria met
- Stop condition:
  - stop if protocol cannot be specified leakage-safely

## J1: Feature Redesign

- Purpose:
  - redesign signal-carrying features before adding new architecture
  - repair positive-delta and hard-error signal
- Allowed changes:
  - DA-relative features
  - residual and history statistics
  - segment-local rolling statistics
  - forecast-side pressure, ramp, PMOS, calendar, and segment features
- Forbidden changes:
  - gate-first retries
  - expert-first retries
  - month routing, fallback policies, or test-driven selectors
- Experiment variants:
  - DA-relative residual features
  - delta history features
  - segment-local rolling stats
  - forecast-side pressure and ramp features
- Per-module tuning loop:
  - start with one feature family at a time
  - keep the backbone fixed
  - accept only if full-year evidence improves
- Pass criteria:
  - at least one feature family improves primary metrics or hard-error diagnostics without leakage risk
- Fail criteria:
  - no feature family improves signal quality after budgeted variants
- Switch criteria:
  - after three completed feature-family failures, switch to J2 strong-baseline redesign with the best clean feature subset
- Output artifacts:
  - feature manifest
  - leakage audit
  - monthly and full-year summaries
  - ledger entries
- Advance condition:
  - accepted feature set exists
- Stop condition:
  - stop J1 if feature additions only improve capped metrics while raw primary metrics degrade consistently

## J2: Strong Baseline

- Purpose:
  - build `SGDFNet-Core-Baseline` on top of accepted J1 features
  - repair direction and hard-error behavior with minimal architecture
- Allowed changes:
  - feature-enhanced delta baseline
  - direction repair
  - segment-conditioned baseline
  - tail-weighted or asymmetric objectives if justified
- Forbidden changes:
  - graph modules
  - time-frequency modules
  - hard-regime experts
  - post-hoc selectors
- Experiment variants:
  - stronger A0 baseline
  - A1 direction repair
  - segment-conditioned baseline
  - tail-weighted objective
- Per-module tuning loop:
  - one baseline mechanism at a time
  - compare against current accepted Protocol B baseline
- Pass criteria:
  - improved full-year `rt_smape` or improved `segment_9_16_rt_smape` with no severe collateral regression
- Fail criteria:
  - direction head collapse persists
  - tail treatment improves only diagnostics but not price metrics
- Switch criteria:
  - after three failed J2 branches, move to J3 only if a strong baseline still exists and the evidence says architecture, not data contract, is the bottleneck
- Output artifacts:
  - protocol-complete run artifacts
  - comparison report against current accepted baseline
  - diagnostics summary
- Advance condition:
  - `SGDFNet-Core-Baseline` accepted
- Stop condition:
  - stop if no stronger baseline can beat the accepted J1/J2 reference within budget

## J3: Time-Frequency Decoupling

- Purpose:
  - test whether simple decomposition helps separate slow anchor drift from sharp residual dynamics
- Allowed changes:
  - moving-average decomposition
  - low/high-frequency residual split
  - only later, learnable decomposition if simpler variants fail
- Forbidden changes:
  - attention-heavy decomposition
  - simultaneous graph introduction
  - multiple decomposition families in one experiment
- Experiment variants:
  - fixed moving-average split
  - low/high-frequency dual-path residual model
  - learnable decomposition only after fixed splits fail
- Per-module tuning loop:
  - start with the smallest decomposition
  - keep the J2 baseline otherwise fixed
- Pass criteria:
  - improved tail or segment metrics with stable full-year primary metrics
- Fail criteria:
  - decomposition adds complexity with no hard-regime gain
- Switch criteria:
  - after budget exhaustion with no gain, switch to J4 or J5 based on diagnostics
- Output artifacts:
  - decomposition config
  - module ablation summary
  - ledger entries
- Advance condition:
  - one decomposition module passes
- Stop condition:
  - stop if decomposition benefit is not reproducible across months

## J4: Graph Structure

- Purpose:
  - test lightweight feature-group interaction modeling
- Allowed changes:
  - feature-group graph only
  - static graph first
  - fixed group taxonomy:
    - DA
    - delta history
    - load
    - renewables
    - PMOS or pressure
    - calendar or segment
    - residual stats
- Forbidden changes:
  - learnable dynamic graph before static graph evidence
  - graph plus new decomposition in the same first-pass experiment
- Experiment variants:
  - static feature-group graph encoder
  - graph residual fusion on top of J2 or J3 baseline
- Per-module tuning loop:
  - hold feature set fixed
  - hold loss fixed
  - vary only graph integration
- Pass criteria:
  - graph improves hard-regime or segment performance without raw-metric collapse
- Fail criteria:
  - graph gains are decorative or vanish outside a small month subset
- Switch criteria:
  - if static graph fails, postpone dynamic graph and move to J5 or J6
- Output artifacts:
  - graph taxonomy note
  - ablation report
  - diagnostics
- Advance condition:
  - accepted graph module exists
- Stop condition:
  - stop graph exploration if static graph fails and no clear interaction hypothesis remains

## J5: Hard-Regime / Spike Correction

- Purpose:
  - add explicit correction only after stronger baseline signal exists
- Allowed changes:
  - leakage-safe hard-regime labels justified by feasibility evidence
  - signed correction only if diagnostics justify it
  - residual or additive correction modules
- Forbidden changes:
  - blind reuse of failed top-absolute-delta labels
  - gate-first retries without new label or feature evidence
- Experiment variants:
  - corrected hard-regime labels
  - signed residual correction
  - asymmetric correction branches
- Per-module tuning loop:
  - first validate labels
  - then validate correction branch
  - then validate signed split only if needed
- Pass criteria:
  - meaningful hard-regime gain on tail or segment metrics plus interpretable diagnostics
- Fail criteria:
  - collapse, near-zero correction, or no price gain
- Switch criteria:
  - if hard-regime labels remain weak, switch to J6 rather than repeating gate tuning
- Output artifacts:
  - label audit
  - correction diagnostics
  - tail error report
- Advance condition:
  - accepted hard-regime correction module exists
- Stop condition:
  - stop if correction only improves direction while price tail stays flat

## J6: Interval / Probabilistic Output

- Purpose:
  - test whether calibrated uncertainty helps spike sensitivity and robustness
- Allowed changes:
  - quantile outputs
  - spike-probability heads
  - heteroscedastic outputs
  - calibration only if leakage-safe under Protocol B
- Forbidden changes:
  - calibration fit on target month
  - probabilistic output without interval evaluation
- Experiment variants:
  - quantile regression
  - uncertainty-weighted residual output
  - spike-probability auxiliary output
- Per-module tuning loop:
  - start with the smallest probabilistic head
  - hold the accepted backbone fixed
- Pass criteria:
  - improved calibration or tail robustness without damaging primary price metrics
- Fail criteria:
  - uncertainty outputs are uncalibrated and do not improve useful metrics
- Switch criteria:
  - after two failed probabilistic variants, stop J6 and proceed to J7 with the best deterministic stack
- Output artifacts:
  - coverage table
  - calibration diagnostics
  - monthly summaries
- Advance condition:
  - one interval or probabilistic module passes
- Stop condition:
  - stop if interval outputs only add complexity without improving decision-relevant metrics

## J7: Full Fusion and Controlled Tuning

- Purpose:
  - combine only previously accepted modules into one controlled candidate
- Allowed changes:
  - fusion of accepted J2, J3, J4, J5, and J6 modules
  - limited tuning of interaction weights or branch dimensions
- Forbidden changes:
  - adding new architecture ideas that skipped module acceptance
  - month routing or hidden fallback logic
- Experiment variants:
  - accepted-module fusion
  - controlled interaction tuning
- Per-module tuning loop:
  - add one accepted module at a time to the current best fused baseline
  - verify additive value before the next addition
- Pass criteria:
  - fused candidate beats best accepted standalone baseline on full-year primary metrics
- Fail criteria:
  - fused stack regresses due to incompatibility
- Switch criteria:
  - if fusion degrades, revert to the strongest simpler package and stop adding modules
- Output artifacts:
  - fusion ablation table
  - best-model comparison
  - final candidate summary
- Advance condition:
  - one fused candidate accepted
- Stop condition:
  - stop if fusion complexity outweighs measurable benefit
